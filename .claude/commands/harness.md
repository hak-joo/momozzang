# /harness — 장기 실행 에이전트 하네스

당신은 이 워크플로우의 **오케스트레이터**입니다. 직접 기획/구현/평가하지 않고, 세 전문 에이전트를 순서대로 호출하며 파일 기반으로 통신시킵니다. 설계 배경은 `.harness/README.md`를 참고하세요.

요청: $ARGUMENTS

## 0. 준비

1. 요청에서 짧은 영문 슬러그를 만든다 (예: "guestbook-search").
2. 실행 디렉토리 `.harness/runs/<슬러그>/`를 만든다.
3. `.harness/runs/<슬러그>/HANDOFF.md`가 이미 존재하면 **재개 모드**다: HANDOFF.md를 읽고, 거기 기록된 단계부터 이어간다. 없으면 1단계부터 시작한다.
4. `PROGRESS.md`를 실행 디렉토리에 만들고, 이후 모든 단계 전환마다 한 줄씩 추가한다 (형식: `- [스프린트N] <단계>: <결과 요약>`).

## 1. 기획 (Planner)

`harness-planner` 에이전트를 호출한다. 프롬프트에 포함할 것:
- 사용자 요청 원문 ($ARGUMENTS)
- 실행 디렉토리 절대 경로
- "SPEC.md를 `.harness/templates/SPEC.md` 템플릿에 따라 작성하라"

완료 후 SPEC.md가 생성됐는지 확인한다. 사용자 요청이 이미 매우 구체적인 작은 수정이라면 이 단계를 건너뛰고 요청 원문을 SPEC.md로 직접 정리해도 된다.

## 2. 스프린트 루프 (Generator ↔ Evaluator)

스프린트 번호 N=1부터 시작. 각 스프린트는 다음 4단계:

### 2-1. 계약 제안
`harness-generator`를 호출해 `SPRINT_CONTRACT_<N>.md` 초안을 작성하게 한다. 직전 `QA_FINDINGS_<N-1>.md`가 있으면 경로를 전달하고 결함 수정을 계약에 포함하게 한다.

### 2-2. 계약 검토
`harness-evaluator`를 호출해 계약의 성공 기준이 검증 가능한지 검토하게 한다. 평가자가 수정을 요구하면 생성자에게 되돌린다 (최대 2회 왕복, 그 이상이면 오케스트레이터가 중재해 확정).

### 2-3. 구현
`harness-generator`를 호출해 합의된 계약을 구현하게 한다. 프롬프트에 실행 디렉토리 경로와 계약 파일명을 명시한다.

### 2-4. QA
`harness-evaluator`를 호출해 능동적 테스트(개발 서버 + Playwright)를 수행하고 `QA_FINDINGS_<N>.md`를 작성하게 한다.

### 분기
- **통과**: SPEC.md의 완료 정의를 점검한다. 남은 기능이 있으면 N+1 스프린트로, 없으면 3단계로.
- **실패**: N+1 스프린트로 진행하되 결함 수정을 최우선 항목으로 한다.
- **동일 결함이 3개 스프린트 연속 미해결**: 루프를 멈추고 사용자에게 상황과 선택지를 보고한다.

## 3. 종료

1. `PROGRESS.md`에 최종 요약을 기록한다.
2. 사용자에게 보고한다: 구현된 기능, 스프린트 수, 최종 QA 판정, 남은 알려진 이슈, 커밋 목록.
3. 커밋은 스프린트마다 생성자가 이미 했으므로, push는 사용자가 요청할 때만 한다.

## 오케스트레이터 규칙

- 에이전트 간 통신은 **파일로만** 한다. 에이전트의 최종 메시지는 진행 판단용으로만 쓰고, 다음 에이전트에게는 항상 파일 경로를 전달한다.
- 컨텍스트가 길어지면 요약되더라도 작업을 이어갈 수 있도록, 매 단계 전환 시 `HANDOFF.md`를 `.harness/templates/HANDOFF.md` 형식으로 갱신한다.
- 생성자의 "통과" 주장을 그대로 믿지 않는다. 합격 판정은 항상 평가자의 QA_FINDINGS 파일 기준이다.
- 스프린트가 비정상 종료(에이전트 실패 등)되면 같은 단계를 1회 재시도하고, 그래도 실패하면 HANDOFF.md를 갱신한 뒤 사용자에게 보고한다.
