# /commit — 브랜치 자동 생성 + 원자적 커밋

사용자 힌트: $ARGUMENTS

## 1. 변경사항 분석

1. `git status --short`와 `git diff` / `git diff --stat`(staged 포함)으로 전체 변경을 파악한다.
2. 변경을 **논리 단위**로 분류한다 (기능 추가 / 버그 수정 / 문서 / 설정 등). 힌트($ARGUMENTS)가 있으면 그 범위만 커밋하고 나머지는 남겨둔다.
3. 커밋 전 점검:
   - `.env` 등 비밀값 파일이 staged 되지 않았는지 확인한다. 비밀값(토큰, 키, URL 값)이 diff에 보이면 커밋을 중단하고 사용자에게 보고한다.
   - gitignore 대상인데 추적 중인 파일이 있으면 보고한다.
   - 의도와 무관한 파일(빌드 산출물, 로컬 상태)을 끼워넣지 않는다.

## 2. 브랜치 결정 (자동)

1. 현재 브랜치를 확인한다 (`git branch --show-current`).
2. **main(또는 기본 브랜치)에 있으면** 변경 내용에 맞는 브랜치를 자동 생성한다:
   - 네이밍: `<타입>/<영문-슬러그>` — 타입은 변경 성격에 따라 `feat` / `fix` / `docs` / `chore` / `refactor` 중 선택 (이 저장소의 기존 브랜치 스타일: `feature/admin`, `docs/project-docs-and-agents` 참고 — `feature/` 접두어도 허용)
   - 예: `feat/gallery-lazy-load`, `fix/account-font-size`
   - `git checkout -b <브랜치명>`으로 생성 후 진행한다.
3. **이미 작업 브랜치에 있으면** 그대로 사용한다. 단, 이번 변경이 현재 브랜치의 주제와 명백히 무관하면 사용자에게 새 브랜치 생성 여부를 확인한다.

## 3. 커밋 생성

1. 논리 단위별로 나눠 커밋한다 (1~3개 권장, 한 커밋 = 한 관심사).
2. 커밋 메시지는 **이 저장소의 기존 스타일**을 따른다: `feat:` / `fix:` / `docs:` / `chore:` 접두어 + 한글 설명 혼용. 스코프가 명확하면 `feat(admin):` 형태.
   - `git log --oneline -10`으로 최근 스타일을 확인하고 맞춘다.
3. 커밋 메시지 끝에 다음 푸터를 붙인다:
   ```
   Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>
   ```
4. `git add`는 **명시적 경로**로만 한다 (`git add -A` 금지 — 의도하지 않은 파일 방지).

## 4. 보고

- 생성/사용한 브랜치명, 커밋 해시와 메시지 목록, 커밋하지 않고 남긴 변경(있다면)을 보고한다.
- **push는 하지 않는다.** PR까지 원하면 `/pr`을 안내한다.
